# Al-HissN Shield — AI Coding Agent Instructions

## Core Principle
**Write code like a human senior engineer.** Every change must be readable, maintainable, and scoped precisely. Do not over-engineer. Do not touch code unrelated to the request.

---

## 1. Scope Discipline — Edit Only What Is Asked

- **Modify ONLY the exact files, functions, or lines relevant to the request.**
- **Never refactor, reformat, or rename unrelated code** — even if it looks "messy" or "could be better."
- If a file needs a 3-line change, make a 3-line change. Do not rewrite the file.
- If you need to add a feature, add it in the right place. Do not move existing code around to "make room."
- **Preserve existing patterns, naming conventions, and architecture** unless explicitly asked to change them.

### Examples of what NOT to do:
- ❌ "While I was here, I also cleaned up indentation in 5 other files."
- ❌ "I refactored the entire ViewModel to use a different pattern because it's cleaner."
- ❌ "I renamed these variables to match my personal style."

---

## 2. Readability & Maintainability

### Naming
- Use **descriptive, intention-revealing names**.
- Prefer `isVpnConnected` over `flag`. Prefer `blockMaliciousDomains` over `doStuff`.
- Follow the project's existing naming convention (camelCase, PascalCase, etc.).

### Functions & Classes
- **Single Responsibility**: One function does one thing.
- Keep functions **short and focused** (ideally under 30–40 lines).
- Extract complex logic into **well-named private helpers**.
- Add **KDoc / doc comments** for public APIs and non-obvious logic.

### Comments
- Comment the **"why"**, not the **"what"**.
- Good: `// Coerce to min radius to prevent RadialGradient crash on animation start`
- Bad: `// Set radius to 0.001f`

---

## 3. Architecture & Patterns

- **Respect the existing module structure** (`core-*`, `feature-*`).
- Place new code in the **correct layer**:
  - UI logic → `feature-*/ui/`
  - Business logic → `feature-*/domain/` or ViewModels
  - Data logic → `core-data/`
- Use **dependency injection** (Hilt) consistently.
- Follow **existing Compose patterns** in the project (state hoisting, ViewModel flow, etc.).

---

## 4. Minimal, Predictable Changes

- Before editing, **read the relevant file(s)** to understand context.
- Use `replace_string_in_file` for surgical edits.
- When adding code, place it **logically near related code**.
- **Do not change behavior** that the user didn't ask for.

---

## 5. Testing & Verification

- After changes, **verify the app builds** (`./gradlew assembleDebug`).
- If a bug is fixed, **confirm the fix** (check logs, run on device).
- If UI is changed, **describe what the user should see**.

---

## 6. Communication Style

- Explain **what** you changed and **why**.
- If you made assumptions, state them.
- If a request is ambiguous, **ask for clarification** instead of guessing.
- Keep explanations concise. Show code changes via tools, not pasted blocks.

---

## Summary Checklist Before Submitting

- [ ] Did I touch only the code relevant to the request?
- [ ] Are my variable/function names clear and consistent?
- [ ] Is the code easy to understand without explanation?
- [ ] Did I preserve existing patterns and conventions?
- [ ] Did I verify the change works?
- [ ] Did I explain what I changed and why?
